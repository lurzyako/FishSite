# FishSite: документация проекта

## 1. Назначение проекта

FishSite — учебный интернет-магазин рыбы и морепродуктов под брендом "Морские Дары". Проект объединяет витринный сайт, каталог, корзину, избранное, личный кабинет, оформление заказов, админку, CRM-элементы и backend на FastAPI + PostgreSQL.

Сайт рассчитан на демонстрацию полного пользовательского сценария:

- посетитель смотрит главную страницу и каталог;
- ищет товары, фильтрует категории и открывает карточку товара;
- добавляет товары в корзину и избранное;
- оформляет заказ;
- отслеживает статусы заказов в личном кабинете;
- пишет консультанту в чат;
- администратор управляет заказами, товарами, статусами и чатами.

## 2. Общая архитектура

Проект устроен как единое web-приложение:

- frontend написан на HTML, CSS и vanilla JavaScript;
- backend написан на FastAPI;
- данные хранятся в PostgreSQL;
- FastAPI отдает и API, и статические файлы сайта;
- загрузки изображений товаров сохраняются как файлы, а в базе хранится путь к ним;
- деплой подготовлен для Render, Railway и Procfile-совместимых платформ.

Основной backend-файл — `backend/app.py`. Он создает FastAPI-приложение, подключается к PostgreSQL, инициализирует схему при старте и монтирует статические файлы проекта.

Основной frontend-слой работает через `js/fishsite-core.js`: этот файл отвечает за запросы к API, состояние текущего пользователя, корзину, избранное и общие действия.

## 7. Backend

Backend реализован в `backend/app.py` на FastAPI.

При старте приложение:

1. читает `DATABASE_URL`;
2. подключается к PostgreSQL;
3. применяет `backend/schema.postgres.sql`;
4. применяет `backend/seed.postgres.sql`;
5. проверяет demo-пароли;
6. запускает API;
7. отдает статические файлы проекта.

Адрес документации API при локальном запуске:

```text
http://127.0.0.1:8000/docs
```

### Основные backend-настройки

Переменные окружения:

```bash
DATABASE_URL=postgresql:///fishsite
UPLOAD_DIR=assets/uploads
UPLOAD_PUBLIC_PREFIX=/assets/uploads
```

По умолчанию используется локальная база:

```text
postgresql:///fishsite
```

## 9. Авторизация и сессии

Авторизация работает через cookie `fishsite_session`.

Сценарий:

1. пользователь открывает сайт;
2. backend создает или использует существующую сессию;
3. при входе сессия связывается с `user_id`;
4. frontend получает текущего пользователя через `/api/me`;
5. роль пользователя определяет доступ к админке.

Пароли не хранятся открытым текстом. Используется `PBKDF2-HMAC-SHA256` с солью и большим количеством итераций.

Тестовые пользователи:

```text
Клиент:
email: olga@example.com
password: demo

Администратор:
email: admin@fishsite.local
password: admin
```

OAuth-кнопки Google/VK на frontend предусмотрены, но полноценный backend OAuth пока оформлен как заглушка старта авторизации. Для production нужно добавить callback-эндпоинты, обмен authorization code на access token и привязку внешнего аккаунта к `users`.

## 10. Чат с консультантом

Чат реализован на backend.

### Для гостя

Гость не обязан входить в аккаунт.

Когда неавторизованный пользователь открывает чат:

1. backend создает/использует cookie `fishsite_session`;
2. создает или находит открытый `chat_threads` по `session_id`;
3. сообщения гостя пишутся в `chat_messages` с `sender_role = customer`;
4. в админке такой диалог виден как "Гость сайта";
5. история доступна, пока браузер хранит cookie.

Если гость очистит cookies или откроет сайт в другом браузере, будет создан новый диалог.

### Для авторизованного пользователя

Если пользователь авторизован, диалог связывается с `user_id`, а в админке видны имя и email пользователя.

### Для администратора

В админке есть раздел "Чат".

Администратор может:

- видеть список диалогов;
- видеть счетчик непрочитанных сообщений;
- открывать переписку;
- отвечать клиенту;
- менять статус диалога.

Клиентский чат обновляет сообщения автоматически, пока окно открыто. Админский чат тоже периодически обновляет список и выбранный диалог.

## 11. Каталог и поиск

Каталог загружается через API.

Backend поддерживает:

- получение всех активных товаров;
- серверный поиск;
- синонимы;
- нормализацию русского текста;
- обработку `ё`/`е`;
- упрощенный stemming;
- близкое совпадение для опечаток.

Ключевые синонимы описаны в `SEARCH_SYNONYMS` внутри `backend/app.py`.

Пример:

- "сёмга", "семга", "лосось" считаются близкими запросами;
- "креветки" сопоставляются с "креветка" и `shrimp`;
- "гриль" связан с жаркой, сковородой и запеканием.

## 12. Корзина и оформление заказа

Корзина хранится на backend.

Для гостей корзина связана с `session_id`.

Для авторизованных пользователей корзина может быть связана с `user_id`.

Checkout реализован как последовательность:

```text
Корзина -> Контакты -> Доставка -> Подтверждение
```

При создании заказа:

- создается запись в `orders`;
- создаются строки в `order_items`;
- активная корзина переводится в состояние `converted`;
- создается первая запись в `order_status_history`;
- при наличии пользователя создается уведомление.

## 13. Заказы и статусы

Статусы заказа:

```text
new        — Новый
confirmed  — Подтвержден
processing — Собирается
courier    — Передан курьеру
delivered  — Доставлен
cancelled  — Отменен
```

История статусов хранится отдельно в `order_status_history`.

Это позволяет показывать пользователю не только текущий статус, но и хронологию изменений.

Администратор меняет статус заказа в админке. При изменении статуса backend:

- обновляет `orders.status`;
- добавляет запись в `order_status_history`;
- создает уведомление пользователю, если заказ привязан к аккаунту.

## 14. Админка

Админка находится на странице:

```text
/pages/admin.html
```

Доступ проверяется на frontend через текущего пользователя и роль `admin`.

Основные функции:

- обзор заказов;
- метрики;
- список заказов;
- смена статусов;
- просмотр клиентов;
- просмотр оценок;
- управление товарами;
- загрузка изображений товаров;
- чат с консультантом.

Управление товарами работает через `/api/admin/products`.

Загрузка изображений работает через `/api/admin/products/upload-image`.

Удаление товара в текущей реализации делает товар неактивным, а не удаляет физически из базы. Это безопаснее для истории заказов.

## 16. Запуск локально

### 1. Установить зависимости

```bash
python3 -m pip install -r backend/requirements.txt
```

или через корневой файл:

```bash
python3 -m pip install -r requirements.txt
```

### 2. Создать базу PostgreSQL

```bash
createdb fishsite
```

Если база уже создана в pgAdmin4, этот шаг можно пропустить.

### 3. Указать `DATABASE_URL`, если нужно

Для локальной базы без пароля обычно достаточно значения по умолчанию:

```bash
postgresql:///fishsite
```

Если используется пользователь и пароль:

```bash
export DATABASE_URL="postgresql://USER:PASSWORD@127.0.0.1:5432/fishsite"
```

### 4. Инициализировать базу

```bash
python3 -m backend.init_pg
```

### 5. Запустить сервер

```bash
uvicorn backend.app:app --reload --host 127.0.0.1 --port 8000
```

### 6. Открыть сайт

```text
http://127.0.0.1:8000/
```

API-документация:

```text
http://127.0.0.1:8000/docs
```

## 17. Деплой

Проект можно деплоить как один FastAPI web-service.

### Render

В проекте есть `render.yaml`.

Render создает:

- web service;
- PostgreSQL database;
- переменную `DATABASE_URL`.

Команды:

```text
Build command: pip install -r requirements.txt
Start command: uvicorn backend.app:app --host 0.0.0.0 --port $PORT
```

### Railway

В проекте есть `railway.json`.

Start command:

```bash
uvicorn backend.app:app --host 0.0.0.0 --port $PORT
```

### Procfile

Для Procfile-совместимых платформ:

```text
web: uvicorn backend.app:app --host 0.0.0.0 --port ${PORT:-8000}
```

### Важное про файлы

На бесплатных хостингах файловая система часто временная. Это важно для фото, загружаемых из админки.

Для production лучше:

- подключить persistent disk;
- или использовать S3/CDN;
- в PostgreSQL хранить публичный URL изображения.

## 20. Безопасность и ограничения текущей версии

Что уже сделано:

- пароли хранятся в виде hash;
- используется серверная cookie-сессия;
- админские API требуют роль `admin`;
- товары не удаляются физически, а деактивируются;
- пользовательские данные больше не завязаны на `localStorage` как основное хранилище.

Что стоит улучшить перед production:

- добавить `Secure`, `HttpOnly`, `SameSite`-настройки cookie под production-домен;
- добавить CSRF-защиту для форм и state-changing запросов;
- завершить полноценный OAuth Google/VK;
- добавить rate limiting для авторизации, регистрации, чата и форм;
- добавить валидацию размеров и типов загружаемых файлов;
- подключить внешнее файловое хранилище;
- добавить миграции через Alembic вместо автоприменения SQL-схемы при старте;
- добавить тесты API;
- вынести секреты только в переменные окружения.
